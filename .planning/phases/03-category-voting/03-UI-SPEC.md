## Interaction Contract

### Voting Flow — Player

1. Admin sends `START_CATEGORY_VOTE` via host display button.
2. Player view transitions from lobby section to `.category-vote-section` (add `.active` class, remove from lobby section).
3. Player sees list of ~24 category buttons in default state (no selection).
4. Player taps a category → send `CATEGORY_VOTE` message → button immediately gets `.voted` class (optimistic UI). All other buttons lose `.voted`.
5. Player can re-tap any other category to change their vote — no confirmation required (vote is not destructive).
6. A `.my-vote-label` above the list updates to show the current selection: "Your vote: General Knowledge".
7. When game starts, player view transitions to quiz game section (existing pattern).

### Voting Flow — Host Display

1. Admin sees "Start Category Vote" button in the lobby panel (`.btn-secondary`).
2. On click → `START_CATEGORY_VOTE` sent → button replaced by the `.category-voting-panel`.
3. Each `LOBBY_UPDATE` message updates tally bars in real time (`voteTallies` from room state).
4. The leading category's `.category-tally-card` gets class `.leading` (accent border, pulse animation matches existing `--neon-purple` glow pattern).
5. "Start Quiz" button (`.btn-primary`) is always enabled — admin can start at any time.
6. On start → quiz begins, voting panel replaced by quiz game view (existing `.game-view` pattern).

### Loading / Error States

| State | Surface | Treatment |
|-------|---------|-----------|
| Categories loading (API fetch) | Host display | Replace voting panel body with `.waiting-msg` text "Fetching categories…" with existing `pulse` animation |
| Categories load error | Host display | Show `.toast.toast-error` with message "Couldn't load categories. Try again." — admin can retry by clicking "Start Category Vote" again |
| Player votes when not in voting state | Player | Silent rejection (server returns ERROR, client ignores or shows no feedback — voting UI not visible anyway) |
| No votes cast when admin starts | Both | Quiz launches with first category by alphabetical/lowest-ID order — no special UI needed |

## State Mapping

Map of server state fields (from `room.getState()`) to visual elements:

| State Field | Visual Target | Update Trigger |
|-------------|--------------|----------------|
| `votingActive: true` | Show `.category-vote-section` on player; show `.category-voting-panel` on host | `LOBBY_UPDATE` |
| `availableCategories[]` | Render category list on both surfaces | First `LOBBY_UPDATE` with `votingActive` |
| `voteTallies[categoryId]` | Update `.tally-bar-fill` width and `.tally-vote-count` text on host | Each `LOBBY_UPDATE` |
| Leading category (derived client-side) | Add/remove `.leading` class on host cards | Each `LOBBY_UPDATE` |
| Player's own vote (optimistic) | `.voted` class on player button | Immediately on tap, before server response |

**Tally bar width calculation:**
```js
const maxVotes = Math.max(...Object.values(tallies), 1);
const pct = (tallies[catId] || 0) / maxVotes * 100;
barFill.style.width = pct + '%';
```
Zero-vote bars render at 0% width (not hidden — track is always visible).

## Accessibility

| Requirement | Implementation |
|-------------|---------------|
| Touch target size | Category vote buttons: `min-height: 56px`, full width — exceeds 44px minimum |
| Tap feedback | `.category-vote-btn:active { transform: scale(0.97) }` — consistent with `.btn:active` |

| No color-only information | Vote count numbers shown alongside tally bars — count readable without color |
