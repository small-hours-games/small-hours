The Problem: Infrastructure Before Games
Why the v1.0 roadmap was backwards
▼
The v1.0 roadmap had 12 phases and 83 requirements. The first 5 phases were pure infrastructure: Docker, HTTPS, WebSockets, rooms, lobby. Not a single game was playable until Phase 6.

This is exactly what got the project stuck. We were building plumbing for games that didn't exist yet.

The question that changed everything
Is there any reason this can't start as terminal games talking to a simple API?

Answer: No. JSON in → process → JSON out. That's just functions.
2
Identity Shift: Platform → Engine
PROJECT.md — what this project actually is now
▼
The project name changed from "Small Hours Games" (a platform you deploy) to "Small Hours Game Engine" (a tool for creating games). The core value flipped from "players can join from their phones" to "creating a new game should be trivially simple."
What This Is
-A real-time multiplayer party game platform for local gatherings. One shared
-screen (TV/monitor) acts as the "display" showing game state, and players join
-from their phones via a short room code or QR scan. The platform hosts multiple
-games — all played through a WebSocket-driven Display + Phones architecture.
+A game engine for creating multiplayer party games. JSON in, process, JSON out —
+the engine handles game logic (state, rules, phases, scoring) while remaining
+completely agnostic about how games are presented or how players connect.
Core Value
-Players at a local gathering can instantly join a shared game from their phones
-and play together on a big screen — zero downloads, zero accounts, just a room code.
+Creating a new party game should be trivially simple — define state and rules,
+the engine handles everything else.
What this means
The old core value describes the user experience of the finished product. The new core value describes the developer experience of making games. We build the engine first. The user experience comes from wrapping the engine in a UI later.

6 Constraints → 3
Stripping constraints down to what the engine actually needs
▼
Constraints
- **No auth**: Players are ephemeral — no login, no accounts, by design
-- **Local network optimized**: Designed for same-network play (display + phones)
-- **WebSocket-only real-time**: All game state via WS, HTTP only for management APIs
-- **Docker deployment**: Must run in container with health checks
-- **External API dependency**: Quiz relies on OpenTrivia DB with cache fallback
-- **Mobile-first UI**: Phone controllers must work on all screen sizes with safe areas
+- **JSON boundary**: Engine input/output is always JSON — transport is someone else's problem
+- **Simplicity**: No abstraction before the second use case proves it's needed
The removed constraints aren't wrong — they're just not engine constraints. "Local network optimized" constrains the transport layer. "Docker deployment" constrains ops. "Mobile-first UI" constrains the frontend. None of these constrain (state, action) => newState.

7 SPEC.md Assumptions Challenged
Research findings — what the spec got wrong
▼
Four parallel researchers analyzed the SPEC.md against first principles and existing frameworks (boardgame.io, Colyseus). Every architectural assumption was challenged. All 7 were rejected or made conditional.
SPEC.md AssumptionVerdictInstead
100ms server tick loop REJECT Event-driven: broadcast when state changes. boardgame.io has no tick.
WebSocket-only transport CONDITIONAL Engine is transport-agnostic. WS is one adapter, added later.
Display + Phone roles REJECT Engine has participants + observers. TV vs phone is presentation.
GameController base class REJECT Plain objects with pure functions. Extract patterns after 2-3 games.
Rooms in engine REJECT Session layer concern. Engine has no "room" concept.
30+ typed message protocol REJECT 3 generic types: action, state, event. Games define payloads.
Full state every tick REJECT State on change + events for effects.
Key insight from research
The engine is: (state, action) => {newState, events}

That's it. Pure function. No side effects, no I/O, no transport, no timers. Fully testable. Fully portable. Everything else is a layer above it.

Pitfalls We're Watching For
Over-engineering is one trap — under-engineering is another
▼
"Just build games and extract patterns" sounds simple but has known failure modes. The research identified 7 under-engineering pitfalls specific to these games. Three are critical.

Critical: Private State (affects Shithead, Spy, CAH)

Number Guess and Quiz have no private state — all players see the same thing. If we build the engine assuming that, Shithead (private card hands) forces a rewrite.

Prevention: Engine contract includes view(state, playerId) from day 1. Number Guess returns the same view for everyone — that's fine. But the contract is ready for private state.

Critical: Pattern Extraction Step

Build 3 games with no extraction step → 3 divergent implementations with zero shared code. Game 4 starts from scratch.

Prevention: After game 2 (Quiz), explicitly review both games and extract shared patterns before building game 3.

Critical: Timers as Synthetic Actions

If game code calls setTimeout directly, games become untestable. Timer state tied to wall clock breaks everything.

Prevention: Session layer manages timers. When a timer expires, it sends {type: 'TIMER_EXPIRED'} to the engine like any other action. Engine stays pure.

What Happens Next
The new milestone's build sequence
▼
  1. Define the engine contract — What does a game export? setup(), action(), view(), endIf(). Prove it with Number Guess.
  2. Build terminal client — readline-based, consumes engine JSON, permanent dev tool.
  3. Build Quiz — Adds timers (as synthetic actions), simultaneous play, external data.
  4. Extract patterns — What's shared between Number Guess and Quiz? Extract before game 3.
  5. Build Spy or CAH — Adds roles, private state, turn order.
  6. Build Shithead — Most complex rules. If the engine handles this cleanly, it handles anything.
  7. Add transport layer — WebSocket/HTTP adapters consuming the same JSON engine.
  8. Add session layer — Rooms, player tracking, browser client.
The test
If the engine is right, adding the WebSocket/browser layer in step 7 should require zero changes to any game code. The games don't know and don't care how they're being consumed.
